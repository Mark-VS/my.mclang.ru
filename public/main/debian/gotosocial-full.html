Подготовка Nginx и установка сертификата

У меня в данный момент четыре адреса в dns-записях:

  • mclang.ru
  • my.mclang.ru
  • gts.mclang.ru
  • test03.mclang.ru

Нам нужно установить сертификат для этих адресов, чтобы можно было подключаться к ним через протокол HTTPS. Но для начала давайте создадим минимальный конфиг для Nginx, который будет работать с HTTP. Напишем "Hello, world!" для адреса test03.mclang.ru.

# /etc/nginx/sites-available/test03.mclang.ru

server {
   listen 80;
   listen [::]:80;
   server_name    test03.mclang.ru www.test03.mclang.ru;
   location / {
      default_type "text/plain; charset=utf-8";
      return 200 "Привет, тупица!";
   }
}

Создаём в папке sites-enabled ссылку на наш конфиг:

sudo ln -s /etc/nginx/sites-available/test03.mclang.ru /etc/nginx/sites-enabled/

# Проверим конфиг на ошибки:
sudo nginx -t

# Запускаем Nginx:
sudo systemctl start nginx

Теперь попробуем зайти на страничку по адресу: http://test03.mclang.ru. Если всё работает, то браузер должен показать нам надпись: "Привет, тупица!". Если всё в порядке, идём дальше.

Прежде, чем создавать сертификат, возможно, нам понадобится проверить список уже имеющихся сертификатов и удалить какой-нибудь из них.

# Посмотреть список сертификатов:
sudo certbot certificates

# Удалить сертификат можно вот так:
sudo certbot delete --cert-name test01.mclang.ru

# Или вот так:
sudo certbot delete
   # а далее выбираем из предложенных сертификат,
   # который хотим удалить

Теперь мы создаём сертификат для наших четырёх адресов.

Важно! Мы создаём протокол через python3-certbot-nginx, поэтому следующую команду выполняем с запущенным Nginx:

sudo certbot --nginx -d mclang.ru -d www.mclang.ru \
    -d my.mclang.ru -d www.my.mclang.ru \
    -d gts.mclang.ru -d www.gts.mclang.ru \
    -d test03.mclang.ru -d www.test03.mclang.ru

Сертификат создался. Правда certbot ругнулся на то, что он не нашёл блоков для остальных имён, кроме test03.mclang.ru и на то, что он не смог автоматически модифициорвать конфиг, но нам это и не нужно.

Сертификат появился в папке /etc/letsencrypt/mclang.ru/. Название папки mclang.ru взялось из-за того, что именно этот домен я указал первым в списке.

Теперь добавим в конфиг Nginx'а второй блок – для протокола HTTPS:

# /etc/nginx/sites-available/test03.mclang.ru

# Первый блок оставляем как есть
server {
   listen 80;
   listen [::]:80;
   server_name    test03.mclang.ru www.test03.mclang.ru;
   location / {
      default_type "text/plain; charset=utf-8";
      return 200 "Привет, тупица!";
   }
}

# А этот добавляем:
server {
   listen 443 ssl;
   listen [::]:443 ssl;
   server_name   test03.mclang.ru www.test03.mclang.ru;
   ssl_certificate /etc/letsencrypt/live/mclang.ru/fullchain.pem;
   ssl_certificate_key /etc/letsencrypt/live/mclang.ru/privkey.pem;
   location / {
      default_type "text/plain; charset=utf-8";
      return 200 "Это безопасная строка, полученная через протокол HTTPS.";
   }
}

После изменения конфига нужно обязательно перезапустить Nginx. Пока Nginx не перезапущен, он будет продолжать работать со старыми настройками.

# Проверяем конфиг на работоспособность:
sudo nginx -t

# И если всё в порядке, перезагружаем Nginx:
sudo systemctl restart nginx

Теперь можно открывать страничку по адресу https://test03.mclang.ru.

http://test03.mclang.ru – здесь мы увидим надпись "Привет, тупица!".

https://test03.mclang.ru – а здесь появится надпись: "Это безопасная строка, полученная через протокол HTTPS".

Таким образом у нас два блока: один отвечает за HTTP-запросы, а другой за HTTPS-запросы. Мы можем писать для них разные версии сайта и в зависимости от того, какой протокол ввёл пользователь в строку бразуера, отдавать разный результат. Но обычно в блоке с HTTP-протоколом просто делают редирект на 443-й порт. Таким образом сайт всегда будет работать только через безопасный HTTPS-протокол:

# /etc/nginx/sites-available/test03.mclang.ru

server {
   listen 80;
   listen [::]:80;
   server_name    test03.mclang.ru www.test03.mclang.ru;
   return 301 https://$host$request_uri;
}

# ...
# а секцию с портом 443 оставляем как есть

Если мы просто хотим отдавать статические файлы, то блок с HTTPS-протоколом можно изменить на:

server {
   listen 443 ssl;
   listen [::]:443 ssl;
   server_name   test03.mclang.ru www.test03.mclang.ru;
   ssl_certificate /etc/letsencrypt/live/mclang.ru/fullchain.pem;
   ssl_certificate_key /etc/letsencrypt/live/mclang.ru/privkey.pem;
   root /var/www/my_site;
   location / {
      try_files $uri $uri/ =404;
   }
}

Здесь мы добавили директиву root и подправили location /.

Теперь нам нужно настроить Nginx таким образом, чтобы он выполнял роль прокси-сервера. GoToSocial прослушивает порт 8080 и нам нужно, чтобы Nginx перенаправлял все запросы с порта 443 на 8080:

server {
   listen 80;
   listen [::]:80;
   server_name    test03.mclang.ru www.test03.mclang.ru;
   return 301 https://$host$request_uri;
}

server {
   listen 443 ssl;
   listen [::]:443 ssl;
   server_name   test03.mclang.ru www.test03.mclang.ru;
   ssl_certificate /etc/letsencrypt/live/mclang.ru/fullchain.pem;
   ssl_certificate_key /etc/letsencrypt/live/mclang.ru/privkey.pem;
   location / {
      proxy_pass http://127.0.0.1:8080;

      proxy_set_header Host $host;
      proxy_set_header X-Real-IP $remote_addr;
      proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
      proxy_set_header X-Forwarded-Proto https;

      proxy_redirect off;
   }
}

Есть ещё один нюанс, касающийся размера загружаемых файлов. В GoToSocial он по умолчанию равен 40MB. А вот в Nginx этот размер по умолчанию: 1MB. Поэтому было бы неплохо увеличить максимальный размер загружаемых файлов до 40MB:

# Ставим чуть больше, чем в GoTosocial, ...
# ... чтобы nginx никогда не был узким местом
client_max_body_size 42M;

Осталось проверить конфиг на ошибки и перезагрузить Nginx:

sudo nginx -t
sudo systemctl restart nginx
