Когда я собираю GoToSocial, временные файлы пишутся в /tmp.

А у меня на Debian /tmp – это файловая система tmpfs, т.е. временная фс в оперативной памяти (RAM). То есть /tmp – это не диск, это память компьютера.

lsblk
  # ...
  # tmpfs                450M  274M  176M  61% /tmp
  # ...

Есть переменная TMPDIR, откуда программы узнают о том, куда писать временные файлы. Если эта переменная не задана, то по умолчанию временные файлы пишутся в /tmp. У меня TMPDIR не задана. Поэтому, если я хочу, чтобы Go при сборке писал файлы на диск, то можно просто задать нужный путь в переменной TMPDIR:

export TMPDIR=~/tmp

TLS

Следует обратить внимание на настройку protocol: "https". Она НЕ ВКЛЮЧАЕТ https на сервере, как может показаться на первый взгляд. Она говорит приложению, КАК оно должно считать себя доступным снаружи. То есть, КАК СЕРВЕР ДУМАЕТ О СЕБЕ. Это значит: "мой внешний адрес: https://...". Это нужно, если мы используем Nginx – чтобы генерировать URL-адреса с правильным протоколом. То есть, если Nginx доступен для внешнего мира через https://..., а GoToSocial работает за Nginx'ом через http-протокол, то нужно исправлять URL-адреса, которые сервер отдаёт клиентам, с http://... на https://.... Чтобы GoToSocial с этим справлялся и нужна эта настройка.

А как тогда сказать серверу, чтобы он работал через https?

Для того, чтобы сервер запустился через https, нужно просто указать ему путь к сертификату в tls-certificate-chain и путь к ключу в tls-certificate-key.

То есть, если сертификат указан, то сервер запускается через https, а если сертификат не указан (поля пустые), то сервер запускается через http.

Если оставить настройки как есть и запустить GoToSocial без Nginx, то снаружи он будет работать некорректно. Сертификаты отсутствуют, а значит сервер работает через http, но при этом он генерирует все URL-адреса во внешний мир с https://.... Получается, что все внешние приложения, когда будут подключаться к нам, будут пытаться зайти через https, а у нас сервер запущен на http и получается несоответствие.

Зачем тогда такие настройки выставлены по умолчанию?

Расчитано на то, что GoToSocial запущен за Nginx'ом. Nginx принимает подключения из внешнего мира по https и перенаправляет их GoToSocial, который работает через http. Именно поэтому и порт по умолчанию: 8080, а не дефолтный 80, или 443.

То есть, GoToSocial запущен на порту 8080 и работает через http. А Nginx работает через https и слушает дефолтные порты 443 и 80 и перенаправляет трафик GoToSocial'у на порт 8080.

На такую схему расчитана дефолтная конфигурация GoToSocial.

Nginx

Вот так будет выглядеть конфиг сервера Nginx:

server {
   listen 80;
   listen [::]:80;
   server_name   test02.mclang.ru www.test02.mclang.ru;

   return 301 https://$host$request_uri;
}
server {
   listen 443 ssl;
   listen [::]:443 ssl;
   server_name   test02.mclang.ru www.test02.mclang.ru;
   ssl_certificate /etc/letsencrypt/live/test01.mclang.ru/fullchain.pem;
   ssl_certificate_key /etc/letsencrypt/live/test01.mclang.ru/privkey.pem;

   location / {
     proxy_pass http://127.0.0.1:8080;
     proxy_set_header Host $host;
     proxy_set_header X-Real-IP $remote_addr;
     proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
     proxy_set_header X-Forwarded-Proto https;
   }
}

proxy_set_header ... – это значит: “Когда Nginx пересылает запрос на backend (GoToSocial), добавь/перезапиши HTTP-заголовки”

  • Host – какой домен запросил пользователь

  • X-Real-IP – IP реального пользователя

    Если не указать, все запросы будут выглядеть как 127.0.0.1

  • X-Forwarded-For – цепочка IP

    Это заголовок, который хранит IP-адреса всех, через кого прошёл запрос

    У меня вот такая цепочка: User → Nginx → GoToSocial

    Без X-Forwarded-For GoToSocial увидит: 127.0.0.1, потому что запрос пришёл от Nginx

    X-Forwarded-For "вклеивает" настоящий IP пользователя.

    А $proxy_add_x_forwarded_for означает: “добавь IP пользователя к уже существующей цепочке”

    Если есть только Nginx: User → Nginx → GoToSocial, то X-Forwarded-For: 1.2.3.4

    Если есть ещё прокси/CDN: ser → Cloudflare → Nginx → GoToSocial, то X-Forwarded-For: 1.2.3.4, 5.6.7.8

  • X-Forwarded-Proto https; – говорит бэкенду: пользователь пришёл по https

    Зачем это указывать вручную? Nginx принимает https, а до бэкенда идёт http. Бэкенд сам не видит https.

Запускаем Nginx и перезагружаем GoToSocial. Теперь сервер будет доступен по адресу: https://test02.mclang.ru

systemd service

Создаём файл сервиса:


sudo touch /etc/systemd/system/gotosocial.service

Дефолтный gotosocial.service можно найти в папке с исходниками в ./example/gotosocial.service.

Пример гпт:

# /etc/systemd/system/gotosocial.service

[Unit]
Description=GoToSocial Server
After=network.target
Wants=network-online.target

[Service]
User=gotosocial
Group=gotosocial
ExecStart=/usr/local/bin/gotosocial --config-path /etc/gotosocial/config.yaml server start
Restart=always
RestartSec=5

# безопасность (не обязательно, но хорошо)
# Относится к разделу [Service]
NoNewPrivileges=true
PrivateTmp=true

[Install]
WantedBy=multi-user.target
  • [Unit] – метаданные и зависимости
  • [Service] – поведение процесса
  • [Install]
  • Description=GoToSocial Server – просто описание сервиса

  • After=network.target ‒ Запускай этот сервис после сети

    After= – не означает, что сеть уже готова. Это только порядок запуска, а не гарантия

  • Wants=network-online.target – а вот это зависисмоть

    гарантирует, что сеть реально поднята

WorkingDirectory=/var/lib/gotosocial
