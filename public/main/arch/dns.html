Локальный DNS

  • avahi – только одно имя (hostname)
  • dnsmasq – сколько угодно имён, нормальный DNS, гибкость

avahi

  • avahi – демон, который объявляет мой хост в сети
  • nss-mdns – плагин для glibc (системы резолвинга имён). Он говорит: "Если запрашивают *.local – спроси Avahi"


sudo pacman -S avahi nss-mdns

Далее проверяем /etc/nsswitch.conf.

Там должна быть строка:

hosts: files mdns_minimal [NOTFOUND=return] dns

# А у меня в конфиге вот так:
hosts: mymachines resolve [!UNAVAIL=return] files myhostname dns

Важный нюанс: hostname должен быть задан.

hostnamectl

# Если нужно:
sudo hostnamectl set-hostname gts


# Проверяем, запущен ли демон:
sudo systemctl status avahi-daemon

# Если нет, то запускаем:
sudo systemctl start avahi-daemon

dnsmasq

Для начала установим dnsmasq:

sudo pacman -S dnsmasq

После установки появился конфиг /etc/dnsmasq.conf с очень подробными комментариями, где разъяснены все настройки. Но по умолчанию все они закомментированы.

Добавляем в конец конфига:

# локальный домен
address=/gts.test/192.168.1.3

# И сразу же рекомендуется,
# чтобы dnsmasq не лез в лишнее:
domain-needed
bogus-priv

И запускаем dnsmasq:

sudo systemctl start dnsmasq

Давайте проверим:

# Если я запущу:
dig @127.0.0.1 gts.test
  # dig → 127.0.0.1

# Или:
getent hosts gts.test
  # getent → ::1

gts.test указывает на localhost, а не на 192.168.1.3.

Это потому что у меня в /etc/hosts есть строки, которые перебивают dnsmasq:

127.0.0.1 gts.test
::1       gts.test

В /etc/nsswitch.conf у меня:

hosts: files mdns_minimal ... dns

А значит:

  • сначала /etc/hosts (files)
  • потом DNS

Поэтому в моём случае dnsmasq сейчас вообще не используется.

У меня в файле /etc/nsswitch.conf есть строка: hosts: files mdns_minimal [NOTFOUND=return] dns. Она всё ломает. Из неё нужно убрать "[NOTFOUND=return]". Итоговая строка должна быть такой: hosts: files mdns_minimal dns

Теперь dig получает IP 192.168.1.3, а getent – нет. А всё потому что у меня в /etc/resolv.conf:

nameserver 8.8.8.8
nameserver 8.8.4.4

# Google не знает про gts.test
# Если исправить на:
nameserver 127.0.0.1

... то getent hosts gts.test вернёт нам наш 192.168.1.3

Но, если мы хотим, чтобы у нас работал и интернет, и локальный dns, указанный в dnsmasq, то лучше сделать вот так:

nameserver 127.0.0.1
nameserver 8.8.8.8

# То есть, сначала мы ищем DNS-имена в локальной сети
# И если ничего не найдено, идём на DNS-сервера гугла

Добавим в /etc/dnsmasq.conf ещё три строчки:

# слушаем только нужный интерфейс
interface=enp2s0

# слушаем локалхост и IP в сети
listen-address=127.0.0.1,192.168.1.3

# привязываемся к интерфейсу (важно!)
bind-interfaces

# моя зона
address=/gts.test/192.168.1.3

# полезные опции
domain-needed
bogus-priv

  • interface=enp2s0 – слушать только сетевой интерфейс enp2s0

    Если не указать, то dnsmasq будет слушать все интерфейсы, включая Wi-Fi, VPN, Docker, loopback

  • listen-address=127.0.0.1,192.168.1.3 – на каких IP-адресах принимать запросы

    • 127.0.0.1 – локально на ПК
    • 192.168.1.3 – из сети (телефон, другие устройства)

    Если не указать, dnsmasq может слушать 0.0.0.0 → все IP, включая нежелательные

  • bind-interfaces – заставляет dnsmasq жёстко привязаться к интерфейсу.

    Без этой опции dnsmasq слушает "виртуально". Может перехватывать порты на всех интерфейсах. Может конфликтовать с другими сервисами.

    С этой опцией dnsmasq слушает только указанные интерфейсы и IP, не лезет в другие интерфейсы. Работает предсказуемо.

    bind-interfaces = “не расползайся по системе, работай строго там, где я сказал”

  • domain-needed – не отправлять запросы без домена на внешние DNS-серверы

    То есть, запросы типа localhost, printer или myserver (без точки) не отправляются в интернет (8.8.8.8 и т.п.). А без этой опции получается как бы утечка данных наружу.

  • bogus-priv – не резолвить приватные IP через внешний DNS

    Приватные IP – это следующие диапазоны:

    • 192.168.x.x
    • 10.x.x.x
    • 172.16.x.x – 172.31.x.x

ss (socket statistics) – показывает сетевые соединения и открытые порты

Расшифровка ключей:

  • -t – TCP-соединения
  • -u – UDP-соединения
  • -l – только слушающие (LISTEN). Если не указать флаг -l, то будут показаны все соединения
  • -p – показать процесс (кто владелец)
  • -n – не преобразовывать имена (быстрее)
  • -a – показать все соединения, кроме LISTEN

Сокет – это "точка общения по сети". То есть 192.168.1.3:53 – это сокет. IP + порт.

ss показывает состояния соединений:

  • LISTEN – сервер ждёт подключения
  • ESTAB – соединение активно
  • TIME-WAIT – соединение завершено
  • CLOSE-WAIT – ожидает закрытия
  • UNCONN (unconnected) – такое состояние отображается у UDP, потому что у UDP нет состояния: отправил пакет – забыл. То есть UNCONN для UPD это почти то же, что LISTEN для TCP – сокет готов принимать пакеты, но соединения нет

Посмотреть все слушающие порты:

# Посмотреть все слушающие порты:
ss -ltn

# Посмотреть активные соединения:
ss -tuna

# Посмотреть кто слушает 53-й порт:
ss -tulpn | grep :53

# То же, но отображать только IPv4:
ss -tulpn -4 | grep :53

Утсновить nslookup на Android:

pkg update
pkg install dnsutils

В dnsutils содержится и nslookup, и dig.

nslookup – программа, предназначенная для поиска DNS-записи на сервере: nslookup <имя> <сервер>

Особенность: тут мы явно указываем сервер, поэтому не важно, какой DNS настроен в системе.

Если мы не укажем DNS-сервер явно: nslookup gts.test, то берётся системный. В моём случае первый DNS-адрес из /etc/hosts.

Если я хочу проверить со своего компьютера, работает ли DNS-сервер, который я запустил на этом же компьютере (с помощью dnsmasq), то:

nslookup gts.test 127.0.0.1
  # Server:      127.0.0.1
  # Address:     127.0.0.1#53
  # 
  # Name:        gts.test
  # Address:     192.168.1.3

Если я хочу проверить с Android'а, то:

nslookup gts.test 192.168.1.3
  # Server:     192.168.1.3
  # Address:    192.168.1.3#53
  #
  # Name:       gts.test
  # Address:    192.168.1.3

В моём случае всё нормально: nslookup с Android'а нашёл домен gts.test. Значит, телефон видит dnsmasq, запущенный на ПК.

Но если мы попробуем пропинговать gts.test, то у нас ничего не выйдет, потому что Android в качестве DNS-сервера использует маршрутизатор (192.168.1.1).

Как указать телефону какой DNS-сервер использовать

Есть два способа:

  1. указать DNS-сервер в Android на телефоне
  2. указать DNS-сервер в маршрутизаторе

1. Укажем DNS-сервер в самом телефоне

Для этого заходим в настройки сети. Выбираем "Статический IP" вместо DHCP. И у нас появляются поля DNS 1 и DNS 2.

Далее присваиваем уникальный IP-адрес: 192.168.1.4

В DNS 1 пишем наш 192.168.1.3, а DNS 2 пусть остаётся как есть, чтобы сеть продолжала работать.

Теперь можно пинговать gts.test и всё должно работать:

ping gts.test

2. Укажем DNS-сервер в маршрутизаторе

Нам нужно в Маршрутизаторе указать DNS-сервер: 192.168.1.3

А затем – и это ОЧЕНЬ ВАЖНО! – переподключиться к WiFi. То есть, удалить WiFi-сеть и потом подключиться к ней заново. Важно забыть старые настройки.

И только после этого:

ping gts.test

Любопытно, что если мы запустим nslookup archlinux.org, то увидим гугловский IP:

nslookup archlinux.org
  # Server:       8.8.8.8
  # Address:      8.8.8.8#53
  #
  # Non-Authoritive answer:
  # Name:         archlinux.org
  # Address:      209.126.35.79
  # Name:         archlinux.org
  # Address:      2604:cac0:a104:d::3

При этом ping gts.test тоже пингуется – для его резолвинга используется DNS-сервер 192.168.1.3.

Видимо, в настройках маршрутизатора жёстко прописан Fallback DNS 8.8.8.8 (или в настройках сети на телефоне). И если я пингую gts.test, то используется DNS, который я указал (192.168.1.3), а если я пингую любой другой сайт, например, archlinux.org, то срабатывает Fallback DNS 8.8.8.8.

dig

Команда dig (Domain Information Groper) имеет общий формат:

dig [@сервер] [домен] [тип_записи]

  • @сервер → указываем конкретный DNS-сервер, к которому хотим отправить запрос
  • домен → имя, которое нужно разрешить (например gts.home)
  • тип_записи (необязательный) → например A, AAAA, CNAME, MX и т.д. По умолчанию — A (IPv4-адрес)

dig @192.168.1.3 gts.test
  # ...
  # ; <<>> DiG 9.20.21 <<>> @192.168.1.3 gts.test
  # ; (1 server found)
  # ;; global options: +cmd
  # ;; Got answer:
  # ;; ->>HEADER<<- opcode: QUERY, status: NOERROR, id: 56607
  # ;; flags: qr aa rd ra; QUERY: 1, ANSWER: 1, AUTHORITY: 0, ADDITIONAL: 1
  # 
  # ;; OPT PSEUDOSECTION:
  # ; EDNS: version: 0, flags:; udp: 1232
  # ;; QUESTION SECTION:
  # ;gts.test.                   IN        A
  # 
  # ;; ANSWER SECTION:
  # gts.test.          0         IN        A   192.168.1.3
  # 
  # ;; Query time: 16 msec
  # ;; SERVER: 192.168.1.3#53(192.168.1.3) (UDP)
  # ;; WHEN: Sat Mar 28 22:21:04 MSK 2026
  # ;; MSG SIZE rcvd: 53

Если мы не укажем DNS-сервер: dig gts.test, то будет взят системный из /etc/hosts.

getprop

getprop – это команда Android для чтения системных свойств (system properties).

Выведем все системные свойства, которые содержат dns в имени:

getprop | grep dns
  # [persist.device_config.aconfig_flags.connectivity.data_stall_consecutive_dns_timeout_threshold]: [5]
  # [persist.device_config.aconfig_flags.connectivity.ipclient_accept_ipv6_link_local_dns_version]: [0]
  # [persist.device_config.aconfig_flags.netd_native.dns_event_subsample_map]: [default:6 0:400 2:110 4:110 7:110]

По идее, должны отображаться net.dns1, net.dns2 – основной и вторичный DNS. В моём случае их нет. Отображаются только внутренние флаги Android, которые связаны с DNS. На некоторых телефонах net.dns1/net.dns2 вообще пустые → система использует DoT / встроенный резолвер, а не “прямой” DNS с Wi-Fi.

Как поставить сертификат на Android

У меня на Arch был установлен сертификат с помощью mkcert.

Узнать, где лежит корневой сертификат:

mkcert -CAROOT
  # /home/mark/.local/share/mkcert

Копируем rootCA.pem на Android.

Я его положил в папку My Documents

Теперь нужно установить этот сертификат:

Безопасность и конфиденциальность → Дополнительные настройки → Хранилища учётных данных → Установить сертификаты из памяти → Сертификат центра сертификации → Всё равно установить: termux → storage → shared → My Documents

Посмотреть установленные сертификаты можно так:

Безопасность и конфиденциальность → Дополнительные настройки → Хранилища учётных данных → Доверенные сертификаты → Пользователь

Chrome и Vivaldi доверяют сайту. А вот Firefox нет. И в настройках мобильной версии я не смог найти возможность импортировать сертификат.

На Android Google Chrome использует системный trust store. А другие браузеры используют свой store и не видят системные CA.
