Локальный DNS

Всё про файл /etc/nsswitch.conf: ЗДЕСЬ!

Локальный DNS

  • avahi – только одно имя (hostname)
  • dnsmasq – сколько угодно имён, нормальный DNS, гибкость

avahi

  • avahi – демон, который объявляет мой хост в сети
  • nss-mdns – плагин для glibc (системы резолвинга имён). Он говорит: "Если запрашивают *.local – спроси Avahi"


sudo pacman -S avahi nss-mdns

Далее проверяем /etc/nsswitch.conf.

Там должна быть строка:

hosts: files mdns_minimal [NOTFOUND=return] dns

# А у меня в конфиге вот так:
hosts: mymachines resolve [!UNAVAIL=return] files myhostname dns

Важный нюанс: hostname должен быть задан.

hostnamectl

# Если нужно:
sudo hostnamectl set-hostname gts


# Проверяем, запущен ли демон:
sudo systemctl status avahi-daemon

# Если нет, то запускаем:
sudo systemctl start avahi-daemon
  • Смотрим, какой у нас Static hostname: hostnamectl
  • Я вижу: Govnuter. Значит, наш компьютер доступен как Govnuter.local. Можно так и пинговать: ping Govnuter.local

Но нам доступно только одно имя: Govnuter.local

dnsmasq

Для начала установим dnsmasq:

sudo pacman -S dnsmasq

После установки появился конфиг /etc/dnsmasq.conf с очень подробными комментариями, где разъяснены все настройки. Но по умолчанию все они закомментированы.

Добавляем в конец конфига:

# локальный домен
address=/gts.test/192.168.1.3

# И сразу же рекомендуется,
# чтобы dnsmasq не лез в лишнее:
domain-needed
bogus-priv

И запускаем dnsmasq:

sudo systemctl start dnsmasq

Давайте проверим:

# Если я запущу:
dig @127.0.0.1 gts.test
  # dig → 127.0.0.1

# Или:
getent hosts gts.test
  # getent → ::1

gts.test указывает на localhost, а не на 192.168.1.3.

Это потому что у меня в /etc/hosts есть строки, которые перебивают dnsmasq:

127.0.0.1 gts.test
::1       gts.test

В /etc/nsswitch.conf у меня:

hosts: files mdns_minimal ... dns

А значит:

  • сначала /etc/hosts (files)
  • потом DNS

Поэтому в моём случае dnsmasq сейчас вообще не используется.

У меня в файле /etc/nsswitch.conf есть строка: hosts: files mdns_minimal [NOTFOUND=return] dns. Она всё ломает. Из неё нужно убрать "[NOTFOUND=return]". Итоговая строка должна быть такой: hosts: files mdns_minimal dns

Теперь dig получает IP 192.168.1.3, а getent – нет. А всё потому что у меня в /etc/resolv.conf:

nameserver 8.8.8.8
nameserver 8.8.4.4

# Google не знает про gts.test
# Если исправить на:
nameserver 127.0.0.1

... то getent hosts gts.test вернёт нам наш 192.168.1.3

Добавим в /etc/dnsmasq.conf ещё три строчки:

# слушаем только нужный интерфейс
interface=enp2s0

# слушаем локалхост и IP в сети
listen-address=127.0.0.1,192.168.1.3

# привязываемся к интерфейсу (важно!)
bind-interfaces

# моя зона
address=/gts.test/192.168.1.3

# полезные опции
domain-needed
bogus-priv

Утсновить nslookup на Android:

pkg update
pkg install dnsutils

В dnsutils содержится и nslookup, и dig.

nslookup – программа, предназначенная для поиска DNS-записи на сервере: nslookup <имя> <сервер>

Особенность: тут мы явно указываем сервер, поэтому не важно, какой DNS настроен в системе.

Если мы не укажем DNS-сервер явно: nslookup gts.test, то берётся системный. В моём случае первый DNS-адрес из /etc/hosts.

Если я хочу проверить со своего компьютера, работает ли DNS-сервер, который я запустил на этом же компьютере (с помощью dnsmasq), то:

nslookup gts.test 127.0.0.1
  # Server:      127.0.0.1
  # Address:     127.0.0.1#53
  # 
  # Name:        gts.test
  # Address:     192.168.1.3

Если я хочу проверить с Android'а, то:

nslookup gts.test 192.168.1.3
  # Server:     192.168.1.3
  # Address:    192.168.1.3#53
  #
  # Name:       gts.test
  # Address:    192.168.1.3

dig

Команда dig (Domain Information Groper) имеет общий формат:

dig [@сервер] [домен] [тип_записи]

  • @сервер → указываем конкретный DNS-сервер, к которому хотим отправить запрос
  • домен → имя, которое нужно разрешить (например gts.home)
  • тип_записи (необязательный) → например A, AAAA, CNAME, MX и т.д. По умолчанию — A (IPv4-адрес)

dig @192.168.1.3 gts.test
  # ...
  # ; <<>> DiG 9.20.21 <<>> @192.168.1.3 gts.test

getprop

getprop – это команда Android для чтения системных свойств (system properties).

Выведем все системные свойства, которые содержат dns в имени:

getprop | grep dns
  # [persist.device_config.aconfig_flags.connectivity.data_stall_consecutive_dns_timeout_threshold]: [5]
  # [persist.device_config.aconfig_flags.connectivity.ipclient_accept_ipv6_link_local_dns_version]: [0]
  # [persist.device_config.aconfig_flags.netd_native.dns_event_subsample_map]: [default:6 0:400 2:110 4:110 7:110]

По идее, должны отображаться net.dns1, net.dns2 – основной и вторичный DNS. В моём случае их нет. Отображаются только внутренние флаги Android, которые связаны с DNS. На некоторых телефонах net.dns1/net.dns2 вообще пустые → система использует DoT / встроенный резолвер, а не “прямой” DNS с Wi-Fi.
